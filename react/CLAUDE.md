# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project state

This is a **template repo** (originally generated via `create-tsrouter-app`, see `.cta.json`) meant to be copied/forked as the starting point for other projects — not a finished app. It's a client-only SPA (Vite + React 19 + TanStack Router; no TanStack Start/SSR). It ships one established pattern — Entra ID auth via MSAL — plus a couple of routes as usage examples. When extending it for a real project:
- Follow the existing conventions (route structure, path aliases, context wiring) rather than introducing new ones.
- Treat `src/routes/index.tsx`, `src/routes/_authenticated/account.tsx`, and `src/routes/posts.tsx` as disposable example routes, not fixed requirements — `posts.tsx` specifically demonstrates the data-fetching layering convention (see Architecture below), not a real feature.
- Treat `.env.example` values as placeholders — a project cloning this template must supply its own Entra ID app registration (`VITE_ENTRA_CLIENT_ID`, `VITE_ENTRA_TENANT_ID`, `VITE_ENTRA_REDIRECT_URI`).

## Commands

Package manager is **pnpm** (see `pnpm-workspace.yaml`).

- `pnpm dev` — start dev server on port 3000
- `pnpm build` — production build (runs route generation via the Vite plugin automatically)
- `pnpm preview` — preview a production build
- `pnpm test` — run all tests with Vitest
- `pnpm test -- <pattern>` — run a subset of tests (Vitest CLI filtering, e.g. `pnpm test -- src/foo.test.tsx`)
- `pnpm lint` / `pnpm format` / `pnpm check` — ESLint / Prettier+ESLint fix / Prettier check-only
- `pnpm generate-routes` — manually regenerate `src/routeTree.gen.ts` (normally auto-generated by the router Vite plugin on dev/build; run manually if route files changed and the generated tree looks stale)

## Architecture

- **File-based routing via TanStack Router**: routes live in `src/routes/`. Each route file exports a `Route` created with `createFileRoute('/path')({ component, loader, ... })`. The router plugin (`@tanstack/router-plugin/vite`, configured in `vite.config.ts`) scans this directory and generates `src/routeTree.gen.ts` — never hand-edit that generated file (it's marked read-only/excluded in `.vscode/settings.json`).
- **Root layout**: `src/routes/__root.tsx` defines `createRootRouteWithContext<RouterContext>()` (context: `{ queryClient, auth }`) and renders `<Outlet />` plus the TanStack Devtools panel, with both the router devtools and the query devtools (`src/integrations/tanstack-query/devtools.tsx`) registered in its `plugins` array.
- **Router setup — single entry point**: `src/router.tsx` exports `getRouter()`, which builds `{ queryClient }` via `getContext()` (`src/integrations/tanstack-query/root-provider.tsx`) plus an `auth: undefined!` placeholder, then calls `createTanStackRouter`. `src/main.tsx` calls `getRouter()` once at module scope and mounts to `#app`; the live auth object is injected per-render via `<RouterProvider router={router} context={{ auth }} />` (using `useAuth()`), so the router itself is never recreated on auth changes.
- **TanStack Query — wired, no SSR integration**: `root-provider.tsx` holds a module-level `QueryClient` singleton; `getContext()` returns it for router context (`{ queryClient }`) and the default-exported `TanstackQueryProvider` wraps the app in `<QueryClientProvider client={queryClient}>` (mounted in `main.tsx`, inside `AuthProvider`). Same instance both places, so router loaders and component hooks (`useQuery`/`useSuspenseQuery`) share one cache. `setupRouterSsrQueryIntegration` is intentionally not used — there's no SSR here.
- **Data fetching — three-layer convention, see `src/api/posts.*` as the reference example**:
  1. **API layer** (`src/api/posts.ts`): a plain async function per resource (`fetchPosts`) that does the actual `ky` call and returns typed data. No React Query or router imports here.
  2. **Query layer** (`src/api/posts.keys.ts` + `src/api/posts.queries.ts`): a query-key factory (`postsKeys.all` / `postsKeys.lists()`, extend with `.detail(id)` etc. as needed) and a `queryOptions()`-returning function (`postsQueryOptions()`) that pairs the key with the API layer's fetch function.
  3. **Route layer** (`src/routes/posts.tsx`): `loader` calls `queryClient.ensureQueryData(postsQueryOptions())` (prefetches on `intent`, per `defaultPreload: 'intent'` / `defaultPreloadStaleTime: 0` in `router.tsx`), and the component reads the same options via `useSuspenseQuery` — no separate loading state needed since the loader guarantees the data is already cached.
  - New resources should follow this same `<resource>.ts` / `<resource>.keys.ts` / `<resource>.queries.ts` split under `src/api/`, not one big file.
  - **Two ky clients, don't mix them up**: `src/api/posts.ts`'s client is unauthenticated, for public/third-party APIs. `src/api/authenticated-client.ts` is the pattern for *your own* protected API — it attaches a token via `getAccessToken()` in `beforeRequest`, and on a `401` force-refreshes the token and retries once (`retry.limit: 1` + `beforeRetry`) before giving up. **Never point the authenticated client at a third-party host** — the token's audience only matches your own resource server, so sending it elsewhere leaks a valid credential to that host.
- **Path aliases**: both `#/*` and `@/*` map to `./src/*` (see `tsconfig.json` and `imports` in `package.json`).
- **Styling**: Tailwind CSS v4 via `@tailwindcss/vite` plugin, imported once in `src/styles.css` (pulled in by `__root.tsx`). No separate Tailwind config file is needed for v4's Vite plugin setup.
- **Auth (Entra ID via MSAL) — implemented, the reference pattern for this template**: lives under `src/auth/`.
  - `msal-config.ts` — `PublicClientApplication` config; reads `VITE_ENTRA_CLIENT_ID` / `VITE_ENTRA_TENANT_ID` / `VITE_ENTRA_REDIRECT_URI` via `src/env.ts` (zod-validated `import.meta.env`).
  - `msal-instance.ts` — the singleton `msalInstance`, plus `authReady`: a promise that resolves only after both `initialize()` and `handleRedirectPromise()` complete and the active account is set. Anything that checks `msalInstance.getAllAccounts()` must `await authReady` first — checking earlier races with the async redirect-response processing and can wrongly trigger a duplicate `loginRedirect`.
  - `auth-context.tsx` — `AuthProvider`/`useAuth()`, wraps `@azure/msal-react`'s `useMsal`/`useIsAuthenticated` into `{ isAuthenticated, isLoading, user, login, logout }`. Must render under `<MsalProvider>`.
  - `get-access-token.ts` — `getAccessToken({ scopes?, forceRefresh? })` for calling protected APIs (`acquireTokenSilent`, falling back to `acquireTokenRedirect` on `InteractionRequiredAuthError`). `forceRefresh` backs the 401-retry path in `src/api/authenticated-client.ts`.
  - `src/routes/_authenticated.tsx` — pathless layout route guarding everything under `src/routes/_authenticated/`; its `beforeLoad` awaits `authReady` then redirects to Microsoft login via `context.auth.login()` if there's no account. Reminder: `beforeLoad` guards protect the route's UI only, not any server endpoints — not a concern here since this app has no server functions, but keep in mind if this template grows a backend.
  - Mount order in `main.tsx` matters: `<MsalProvider><AuthProvider><App /></AuthProvider></MsalProvider>`, and rendering itself is gated on `authReady` resolving first.
- **Linting**: ESLint config extends `@tanstack/eslint-config` (`eslint.config.js`) with several rules disabled (`import/no-cycle`, `import/order`, `sort-imports`, `@typescript-eslint/array-type`, `@typescript-eslint/require-await`, `pnpm/json-enforce-catalog`).

## Skill Loading

- Monorepos: when working across packages, run the skill check from the workspace root and prefer the local skill for the package being changed.
- Multiple matches: prefer the most specific local skill for the package or concern you are changing; load additional skills only when the task spans multiple packages or concerns.
